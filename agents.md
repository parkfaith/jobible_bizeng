# jobible_bizeng / 조바이블 비즈영어 기획 메모

이 파일은 `jobible_bizeng` 앱 기획을 다른 컴퓨터나 새 AI 세션에서도 이어가기 위한 기준 문서이다.

새 세션에서는 반드시 이 파일을 먼저 읽고 진행한다.  
기획이 바뀌면 항상 이 파일을 업데이트한다.  
현재는 제품 기획 단계이며, 기능 구현은 아직 하지 않는다.

문서와 화면 예시 파일은 모두 UTF-8 인코딩으로 유지한다.  
PowerShell에서 한글이 깨져 보이면 파일 자체 문제로 단정하지 말고, UTF-8로 읽히는지 먼저 확인한다.

프로젝트 저장소:

- https://github.com/parkfaith/jobible_bizeng.git

## 1. 앱 한 줄 정의

`Jobible BizEng`는 외국계 이직을 준비하는 시니어 AI/IT 리더가 모바일에서 실제 영어 면접처럼 AI와 음성으로 대화하고, 피드백을 받은 뒤 다시 말해 보는 실전형 비즈니스 영어 면접 코치이다.

## 2. 핵심 사용자

대표 사용자: `ryanPark`

- 50세
- AI 기술이사
- 국내 기업에서 20년 이상 근무
- 실무 경험과 조직 리더십 경험은 충분함
- 일상 영어는 어느 정도 가능하지만, 영어 면접에서 경력/성과/리더십을 구조적으로 설명하는 데 약함
- 목표 포지션: AI Director, Head of Engineering, AI Deployment Lead, Enterprise AI PM

이 앱은 영어 초보나 학생용 앱이 아니다.  
경험 많은 시니어 직장인이 외국계 면접에서 떨어지지 않을 자신감을 만드는 앱이다.

## 3. 해결하려는 문제

사용자는 실력과 경력은 충분하지만 영어 면접에서 아래 문제를 겪는다.

- 자기소개와 경력 설명이 길고 흐릿해짐
- 리더십 경험과 프로젝트 성과를 설득력 있게 말하지 못함
- 기술 설명을 영어로 할 때 구조가 무너짐
- 꼬리 질문과 압박 질문에 약함
- 면접처럼 이어지는 영어 대화 연습이 부족함

## 4. 현재 제품 방향

당분간 제품을 넓히지 않는다.

초기 목표는 복잡한 영어 학습 플랫폼이 아니라 `모바일 기반 외국계 영어 면접 5~10분 음성 훈련 앱`이다.

가장 중요한 사용 흐름은 다음 하나다.

1. 오늘의 면접 질문을 받는다.
2. 영어로 말해서 답변한다.
3. AI 면접관이 꼬리 질문을 한다.
4. 답변 피드백을 받는다.
5. 개선 답변을 보고 다시 말한다.
6. 좋은 답변과 표현을 내 답변 노트에 저장한다.

기획이 복잡해지면 이 기본 루프로 돌아온다.

## 5. MVP 핵심 기능

초기 MVP는 아래 5개만 만든다고 가정한다.

### 1. 간단한 프로필 입력

긴 이력서를 받지 않는다. 처음에는 아래만 입력받는다.

- 목표 포지션
- 현재 역할과 경력 연수
- 대표 프로젝트 1~2개
- 가장 걱정되는 면접 상황
- 연습 목표: 자기소개, 경력 설명, 리더십, 기술 설명 중 선택

### 2. 오늘의 질문 연습

짧게 한 문항을 연습하는 모드이다.

- 질문 1개 제시
- 사용자가 영어로 답변
- AI가 꼬리 질문 1개 제시
- 답변 피드백 제공
- 개선 답변을 보고 다시 말하기

### 3. 실전 면접 대화 모드

실제 면접처럼 AI 면접관과 영어로 대화를 이어가는 모드이다. MVP에 반드시 포함한다.  
이 모드는 텍스트 채팅이 아니라 `모바일 음성 대화 중심`으로 설계한다.

- `실전 면접 시작` 버튼을 누르면 먼저 오늘의 면접 유형과 예상 질문 방향을 보여준다.
- 사용자가 안내를 확인한 뒤 `면접 시작`을 눌러 실제 음성 연결을 시작한다.
- AI 면접관이 음성으로 인사 후 첫 질문
- 사용자는 영어로 말해서 답변
- 사용자의 답변은 자동으로 텍스트 변환되어 참고용으로 화면에 표시
- AI 면접관은 사용자의 답변에 따라 자연스럽게 다음 질문 또는 꼬리 질문
- 한 세션은 3~4개 질문, 5~10분으로 제한
- 질문 흐름은 오늘의 답변 패턴세트 주제를 중심으로 진행하고, 필요하면 경력/리더십/기술/갈등으로 자연스럽게 확장한다.
- 면접 중에는 코칭하지 않고 실제 면접처럼 진행
- 면접 종료 후 전체 피드백 제공
- 면접 중 사용자가 직접 타이핑하는 UI는 두지 않는다
- 텍스트는 입력 수단이 아니라 참고 자료, 음성 인식 결과, 개선 답변 확인용으로만 사용한다

초기에는 면접관 캐릭터를 여러 개 만들지 않는다.  
`외국계 AI Director 면접관` 1명만 사용한다.

### 4. 답변 코칭

피드백은 많을수록 좋은 것이 아니다. 사용자가 바로 고칠 수 있게 짧고 실용적으로 준다.

일반 답변 피드백:

- 핵심 메시지가 분명한가
- 답변 구조가 좋은가
- 더 자연스러운 영어 표현
- 시니어 리더다운 개선 답변 예시

실전 면접 종료 피드백:

- 가장 좋았던 답변 1개
- 가장 위험했던 답변 1개
- 다음 연습에서 반드시 고칠 점 1개
- 면접에서 바로 쓸 수 있는 개선 문장 3개

### 5. 내 답변 노트

면접 전에 다시 볼 수 있는 답변 저장소이다.

- 자기소개 답변
- 경력 설명 답변
- 리더십 사례 답변
- 기술 프로젝트 설명 답변
- 실패/갈등 해결 답변
- 바로 쓸 수 있는 핵심 영어 문장

## 6. MVP 화면

화면은 3개로 충분하다.

초기 화면은 모바일을 기본 기준으로 설계한다.  
이 앱은 출퇴근 전후, 점심시간, 면접 직전처럼 짧은 시간에 쓰일 가능성이 높으므로 데스크톱보다 모바일 사용성이 중요하다. 데스크톱 화면은 나중에 확장한다.

### 1. 홈

- 오늘의 질문 연습 시작
- 음성 실전 면접 시작
- 최근 저장한 답변
- 이번 주 연습 횟수

### 2. 연습 화면

- AI 면접관 음성 질문
- 큰 마이크 버튼
- 녹음 상태와 남은 시간
- 음성 인식된 답변 텍스트
- 참고용 답변 구조 또는 핵심 표현
- 면접 후 피드백
- 다시 말하기

`오늘의 질문 연습`과 `실전 면접 대화 모드`는 같은 연습 화면 안에서 처리한다.

### 3. 답변 노트

- 저장된 답변
- 저장된 핵심 표현
- 면접 전 빠르게 다시 볼 답변 모음

## 7. 모바일 사용 방향

모바일을 기본 사용 환경으로 본다. 다만 처음부터 네이티브 앱을 만들기보다 `모바일 웹` 또는 `PWA`로 시작하는 것이 현실적이다.

모바일에서 특히 중요한 점:

- 큰 `음성 실전 면접 시작` 버튼
- 한 손으로 누르기 쉬운 녹음/일시정지/제출 버튼
- AI 질문은 짧게 표시하고 음성으로 들려주기
- 사용자의 음성 인식 텍스트는 참고용으로 크게 보여주기
- 타이핑 입력 영역은 두지 않기
- 공부를 돕는 참고 내용은 읽기 전용으로 제공하기
- 면접 중에는 화면 조작을 최소화하기
- 조용한 공간이 아니어도 이어폰으로 연습할 수 있게 고려하기
- 녹음 권한, 마이크 상태, 네트워크 상태를 명확히 보여주기

모바일에서 당장 복잡하게 만들 필요는 없다.  
초기 모바일 경험은 `버튼 누르기 -> 듣기 -> 말하기 -> 음성 인식 결과 확인 -> 다음 질문` 흐름만 자연스러우면 충분하다.

## 8. 초기에는 제외할 기능

좋은 아이디어라도 초기에는 아래 기능을 넣지 않는다.

- 모든 비즈니스 업무 상황 훈련
- 회의, 협상, 고객 커뮤니케이션 전체 시나리오
- 정교한 발음 점수화
- 학습 레벨, 배지, 랭킹, 게임화
- 커뮤니티
- 채용공고 검색
- 이력서/LinkedIn 전체 자동 작성
- 여러 AI 면접관 캐릭터
- 과도하게 세분화된 진도 관리

단, `실전 면접 대화 모드`는 제외하지 않는다. 이 앱의 핵심 경험이다.

## 9. AI 코치의 말투와 역할

AI는 친절한 영어 선생님보다 `실전 면접 코치이자 글로벌 커리어 멘토`처럼 동작해야 한다.

- 지나치게 귀엽거나 가벼운 말투는 피한다.
- 답변을 무조건 칭찬하지 않는다.
- 시니어 리더에게 맞는 간결하고 설득력 있는 표현을 제안한다.
- 면접 중에는 실제 면접관처럼 질문한다.
- 코칭은 면접이 끝난 뒤에 한다.

## 10. 기획 검토 및 수정 제안

현재 기획은 MVP로 적절하다. 다만 아래 사항은 다음 기획에서 반드시 더 구체화한다.

### 1. 실전 면접 대화 모드를 중심 기능으로 둔다

`오늘의 질문 연습`은 가벼운 워밍업이고, 핵심 경험은 `실전 면접 대화 모드`이다.  
홈 화면에서도 `음성 실전 면접 시작`을 가장 중요한 행동으로 배치한다.

### 2. 프로필 입력은 경력 사실 수집에 집중한다

AI가 사용자의 경력을 지어내면 안 된다. 온보딩에서는 멋진 문장보다 실제 사실을 짧게 수집한다.

- 대표 프로젝트명
- 맡은 역할
- 해결한 문제
- 사용한 기술 또는 방법
- 성과 수치 또는 비즈니스 임팩트
- 리더십/갈등/실패 사례 1개

### 3. 면접은 영어로, 코칭은 한국어 중심으로 한다

실전 면접 대화는 영어로 진행한다.  
면접 후 피드백은 사용자가 빠르게 이해할 수 있도록 한국어 중심으로 제공하고, 개선 문장은 영어로 제공한다.

### 4. 피드백 기준을 단순한 4개 항목으로 고정한다

답변 평가는 아래 4개만 본다.

- 내용: 질문에 제대로 답했는가
- 구조: 결론, 근거, 사례, 성과가 보이는가
- 영어: 표현이 자연스럽고 간결한가
- 리더십 톤: 시니어 리더답게 들리는가

점수화가 필요하다면 항목별 5점 척도까지만 사용한다.

### 5. 답변 노트는 최종 답변 저장소로 만든다

답변 노트에는 단순 대화 기록이 아니라 면접 전에 다시 볼 수 있는 최종본을 저장한다.

- 원래 답변
- 개선된 답변
- 최종 암기용 답변
- 바로 쓸 핵심 표현 2~3개

### 6. 압박 질문은 별도 기능으로 만들지 않는다

초기에는 `압박 질문 모드`를 따로 만들지 않는다.  
실전 면접 대화의 후반부에 자연스러운 꼬리 질문이나 검증 질문을 1~2개 넣는 정도로 충분하다.

### 7. 실전 면접은 음성 우선으로 설계한다

이 앱은 면접 훈련 앱이므로 실전 면접 대화는 음성 우선이어야 한다.

- AI 면접관 질문은 음성으로 들려준다.
- 사용자는 영어로 말해서 답변한다.
- 화면에는 음성 인식된 답변 텍스트를 함께 보여준다.
- 음성 인식 결과는 확인용이며, 면접 중 직접 수정하지 않는다.
- 피드백은 면접이 끝난 뒤 제공한다.
- 면접 중 타이핑은 하지 않는다.

초기에는 정교한 발음 점수화나 텍스트 작성 기능보다 `말문이 막히지 않고 면접 흐름을 이어가는 경험`이 더 중요하다.  
구현 단계에서는 최신 공식 OpenAI 문서를 확인한 뒤 음성 입력, 실시간 음성 대화, 녹음 기반 피드백 중 MVP에 맞는 방식을 결정한다.

### 8. 성공 기준을 작게 잡는다

초기 MVP의 성공은 많은 기능이 아니라 아래 행동이 반복되는지로 판단한다.

- 사용자가 실전 면접을 끝까지 완료한다.
- 사용자가 개선 답변을 다시 말한다.
- 사용자가 답변 노트에 최종 답변을 저장한다.
- 일주일에 3회 이상 연습할 이유가 생긴다.

현재 화면 예시는 아래 3개 화면을 포함한다.

- 홈: 오늘의 질문, 음성 실전 면접 시작, 최근 답변, 프로필 요약
- 연습 화면: 모바일 음성 면접, 답변 텍스트 변환, 참고 표현, 면접 후 피드백 예시
- 답변 노트: 원래 답변, 개선 답변, 최종 암기용 답변, 핵심 표현

다음 검토에서는 아래를 확인한다.

- 홈에서 `음성 실전 면접 시작`이 충분히 중심 행동으로 보이는가
- 연습 화면이 실제 음성 면접처럼 긴장감 있게 느껴지는가
- 타이핑 없이도 연습 흐름이 자연스러운가
- 피드백이 너무 많거나 학습 앱처럼 느껴지지 않는가
- 답변 노트가 면접 전날 다시 보기 좋은 구조인가

## 13. API 사용 방향 초안

현재 판단은 아래와 같다.

### 1. 사용자-facing 음성 면접

모바일에서 실제 면접처럼 음성으로 대화하는 핵심 기능은 OpenAI Realtime API 계열을 우선 검토한다.

이유:

- 실시간 음성 입출력에 맞는 구조이다.
- 모바일 웹/PWA에서 WebRTC 기반 연결을 검토할 수 있다.
- 사용자의 말투, 끊김, 답변 흐름을 살린 면접 경험에 적합하다.

### 2. 면접 후 피드백과 답변 노트

면접이 끝난 뒤에는 음성 대화 기록 또는 전사 텍스트를 바탕으로 별도 텍스트 모델이 피드백을 생성한다.

피드백 생성 항목:

- 내용
- 구조
- 영어 표현
- 리더십 톤
- 최종 암기용 답변
- 바로 쓸 핵심 표현

### 3. 매일 제공할 면접영어 패턴 콘텐츠

매일 받는 면접영어 패턴, 질문 세트, 모범 답변, 핵심 표현은 실시간 음성 API가 아니라 텍스트 생성 API로 만드는 것이 적합하다.

이 영역은 Anthropic Claude API도 적극 비교한다.  
Claude Code 자체는 개발 보조 도구이지만, Claude 계열 모델은 긴 글 정리, 패턴화, 커리큘럼 구성, 문장 다듬기, 사례 기반 콘텐츠 제작에서 강점이 있을 수 있다.

콘텐츠 생성에서 비교할 항목:

- 시니어 리더에게 맞는 자연스러운 비즈니스 영어 표현
- 면접 질문 패턴의 실전성
- 모범 답변의 구조와 설득력
- 한국어 설명의 명확성
- 매일 반복 학습하기 좋은 분량 조절
- 답변 노트에 저장하기 좋은 최종 문장 품질

### 4. 현재 추천

MVP 단계 추천은 `음성은 OpenAI, 콘텐츠는 OpenAI와 Claude 비교` 구조이다.

- 음성 면접: OpenAI Realtime API
- 면접 후 피드백: 우선 OpenAI 텍스트 모델, 이후 Claude와 품질 비교
- 매일 면접영어 패턴 생성: OpenAI와 Claude API를 같은 프롬프트로 비교
- 최종 선택 기준: 비용보다 `콘텐츠 품질`, `일관성`, `운영 단순성`을 우선한다

Claude Code는 개발 보조 도구로 보고, 실제 앱 사용자에게 제공되는 런타임 API로 보지는 않는다.  
앱에서 Claude를 쓴다면 Claude Code가 아니라 Anthropic Claude API를 사용한다.

구현 단계에서는 최신 공식 OpenAI 문서와 Anthropic 문서를 다시 확인한 뒤 모델명, 비용, 지연시간, 모바일 브라우저 지원 방식을 결정한다.

## 15. 추가 구현 — 오늘의 답변 패턴세트 (2026-05-19)

기존 `오늘의 표현 카드`는 단일 표현 학습에 가까웠으므로, 앱의 핵심 흐름에 맞게 `오늘의 답변 패턴세트`로 확장한다.

방향:

- ChatGPT 스타일의 실전적이고 내용 중심인 패턴세트를 우선한다.
- Claude 스타일의 보기 좋은 정리 구조를 UI와 편집 화면에 흡수한다.
- 두 결과를 사용자에게 동시에 보여주지 않는다.
- 앱 안에서는 하나의 최종 curated 패턴세트만 노출한다.
- 패턴세트는 읽는 콘텐츠가 아니라 `30초 답변 워밍업 -> 말하기 연습 -> 피드백 -> 답변 노트 저장` 흐름의 시작점이다.

현재 구현:

- `GET /api/patterns/daily` — OpenAI Responses API로 오늘의 답변 패턴세트 자동 생성, Turso `daily_patterns`에 `daily_pattern_set`으로 캐시
- `POST /api/patterns/daily` — 오늘의 패턴세트 다시 생성
- `PATCH /api/patterns/daily` — 오늘의 패턴세트 수동 수정 저장
- `/patterns` — 오늘의 답변 패턴세트 상세 보기, 수정, 다시 생성, 말하기 연습 이동
- `/review` — 날짜별 패턴 복습 캘린더와 최근 패턴 목록
- 홈 — `30초 답변 워밍업` 카드로 요약 표시
- `/practice?source=pattern` — 패턴세트 질문과 30초 답변 구조를 참고하며 음성 답변 연습
- `/api/feedback` — 패턴세트 기반 연습 시 오늘의 패턴 활용 여부를 짧게 피드백

Realtime API 메모:

- 예전 Beta 엔드포인트 `/v1/realtime/sessions`와 preview 모델은 사용하지 않는다.
- 현재 구현은 `/v1/realtime/client_secrets`로 ephemeral key를 발급한다.
- 브라우저 WebRTC SDP 교환은 `/v1/realtime/calls`로 수행한다.
- 기본 모델은 `OPENAI_REALTIME_MODEL` 환경변수가 있으면 그 값을 쓰고, 없으면 `gpt-realtime`을 사용한다.
- 실전 면접은 비용 방지를 위해 10분 또는 4개 질문 기준으로 자동 종료한다.
- `/api/patterns/daily`의 다시 생성은 하루 3회로 제한한다.
- OpenAI API 오류 원문은 클라이언트에 그대로 반환하지 않고 서버 로그에만 남긴다.

PWA 메모:

- 앱은 모바일 PWA 설치를 전제로 한다.
- `public/manifest.json`과 `app/layout.tsx`에 앱 이름, 아이콘, apple web app 메타데이터가 설정되어 있다.
- Jobible BizEng 로고 원본은 `public/icons/jobible-bizeng-logo.svg`이다.
- PWA 아이콘은 `public/icons/icon-192.png`, `public/icons/icon-512.png`, maskable 아이콘, `public/apple-touch-icon.png`, `app/favicon.ico`로 생성되어 있다.
- Next.js 16 기본 Turbopack 빌드는 PWA 플러그인의 Webpack 설정과 충돌하므로 `npm run build`는 `next build --webpack`을 사용한다.
- `public/sw.js`와 `public/workbox-*.js`는 빌드 산출물이므로 커밋하지 않고 `.gitignore`에 둔다.

UX 결정:

- 홈의 별도 `오늘의 질문 연습` 카드는 제거한다.
- `오늘의 답변 패턴세트` 안의 질문이 곧 오늘의 질문 연습이다.
- `/practice` 기본 진입도 패턴세트 기반 질문 연습으로 연결한다.
- 순수 일일 질문 생성 API는 남겨두되, 필요할 때 `/practice?source=daily`로만 사용한다.
- 오늘 생성한 패턴세트는 `daily_patterns`에 날짜별로 저장되며, `/review`에서 이전 날짜 패턴을 다시 볼 수 있게 한다.
- 복습은 별도 하단 메뉴 `/review`로 분리한다. `/patterns`는 오늘 패턴 상세와 편집 중심으로 유지한다.
- 답변 노트 카테고리는 고정 분류이므로 가로 스크롤 탭이 아니라 줄바꿈형 그리드로 보여준다.

UX 기준:

- 홈에서는 패턴세트 전체를 길게 보여주지 않고 핵심 표현 3개와 오늘의 질문만 압축 표시한다.
- 상세 화면은 패턴, 의미, 활용 포인트, 자주 틀리는 표현, 쉐도잉, 30초 답변 구조를 제공한다.
- `ChatGPT에 붙여넣기` 흐름은 앱 밖으로 사용자를 보내므로 기본 UX에서 제외한다.
- 실전 면접 중에는 패턴세트를 직접 노출하지 않고, 면접 전 워밍업과 종료 후 피드백에만 연결한다.

## 16. 배포 방향 — Vercel Hobby 1차 배포 판단 (2026-05-20)

현재 판단은 `Vercel Hobby`에 1차 배포해도 된다는 쪽이다.  
다만 무료 플랜의 개인/비상업 사용 제한과 함수 실행 시간/사용량 제한을 전제로 한다.

### 1. Vercel Hobby로 가능한 이유

- 앱은 Next.js App Router 기반이므로 Vercel 배포와 잘 맞는다.
- DB는 Vercel 내부가 아니라 외부 Turso/libSQL을 사용한다.
- OpenAI API 키, Turso 토큰, 앱 비밀번호는 Vercel 환경변수에만 둔다.
- 핵심 음성 면접은 Vercel이 WebSocket 서버나 음성 스트림 중계 서버 역할을 하지 않는다.
- `/api/realtime-token`은 OpenAI Realtime API용 ephemeral token만 발급한다.
- 실제 WebRTC 음성 연결은 브라우저가 OpenAI Realtime API와 직접 연결한다.
- 따라서 Vercel Functions가 WebSocket 서버를 지원하지 않는 제약에는 직접 걸리지 않는다.

### 2. 주의할 점

- Hobby 플랜은 개인/비상업 프로젝트용이다. 혼자 쓰는 MVP나 소규모 개인 테스트에는 적합하지만, 외부 사용자 대상 서비스나 유료화 단계에서는 Pro 전환을 검토한다.
- OpenAI 호출이 들어가는 API 라우트는 응답 시간이 길어질 수 있다.
- 특히 아래 라우트는 Vercel Function 시간 제한을 고려해야 한다.

느릴 수 있는 라우트:

- `app/api/patterns/daily/route.ts`
- `app/api/feedback/route.ts`
- `app/api/feedback/interview/route.ts`
- `app/api/transcribe/route.ts`
- `app/api/expressions/daily/route.ts`
- `app/api/questions/daily/route.ts`

권장:

- 위 라우트에는 필요하면 `export const maxDuration = 60;`을 명시한다.
- OpenAI API 원문 오류는 클라이언트에 그대로 노출하지 않는다.
- 개인용 앱이라도 Vercel 사용량, OpenAI 사용량, Turso 사용량을 주기적으로 확인한다.

### 3. Vercel 환경변수

Vercel Project Settings > Environment Variables에 아래 값을 설정한다.

- `TURSO_DATABASE_URL`
- `TURSO_AUTH_TOKEN`
- `OPENAI_API_KEY`
- `OPENAI_TEXT_MODEL=gpt-4o`
- `OPENAI_REALTIME_MODEL=gpt-realtime`
- `APP_PASSWORD`
- `SESSION_SECRET`
- `ANTHROPIC_API_KEY`는 현재 미사용이며, 향후 콘텐츠 품질 비교 시에만 사용한다.

### 4. Vercel 빌드 설정

- Build Command: `npm run build`
- 현재 `package.json`의 build 스크립트는 `next build --webpack`이다.
- Next.js 16 기본 Turbopack 빌드는 PWA 플러그인의 Webpack 설정과 충돌할 수 있으므로 `--webpack`을 유지한다.
- 개발 서버도 같은 이유로 `npm run dev`가 `next dev --webpack`을 사용한다.
- `middleware.ts`는 사용하지 않는다. Next.js 16 경고 대응으로 인증 보호 로직은 `proxy.ts`에 둔다.

### 5. 배포 후 필수 확인

모바일 실제 기기, 특히 iPhone/Safari 또는 설치형 PWA에서 아래를 확인한다.

1. 로그인
2. 홈 로딩
3. 오늘의 답변 패턴세트 생성
4. 30초 답변 녹음
5. OpenAI 전사
6. 답변 피드백 생성
7. 실전 면접 Realtime 연결
8. 면접 종료 후 종합 피드백
9. 답변 노트 저장
10. 통계 화면 반영
11. PWA 설치와 재접속

### 6. 향후 전환 기준

Vercel Hobby를 유지해도 되는 경우:

- 개인용 MVP
- 사용자 1명 또는 매우 소규모 지인 테스트
- 비상업 사용
- API 호출량이 낮고 사용량 모니터링이 가능한 상태

Pro 또는 별도 백엔드를 검토해야 하는 경우:

- 외부 사용자에게 공개
- 유료화 또는 상업적 사용
- OpenAI 피드백 생성이 60초 제한에 자주 걸림
- 사용량 초과 또는 런타임 로그 부족으로 운영이 어려움
- 서버 측 장시간 작업, 큐, 배치 작업, 오디오 파일 저장이 필요해짐

이전 기획 방향:

현재부터는 기획에서 기술 구현 준비 단계로 넘어간다.  
다만 아직 실제 기능 구현은 시작하지 않고, 기술 선택과 데이터 구조를 먼저 정리한다.

### 1. 저장소 방향

개인용 앱이라도 완전 무DB보다는 경량 DB를 사용하는 것을 추천한다.  
현재 사용자가 기존 사이드프로젝트에서 `Turso`를 사용하고 있으므로, 이 프로젝트도 SQLite 파일 직접 관리보다 `Turso`를 우선 검토한다.

이유:

- 면접 세션 기록이 쌓인다.
- 음성 인식 텍스트와 피드백을 다시 봐야 한다.
- 답변 노트는 저장, 수정, 검색, 분류가 필요하다.
- 매일 제공할 면접영어 패턴 콘텐츠도 캐시하거나 저장할 수 있다.
- Turso는 SQLite/libSQL 계열의 단순함을 유지하면서 모바일 웹/PWA + 개인용 백엔드 구조에 붙이기 좋다.
- 이미 사용 경험이 있으므로 개발 속도와 운영 안정성 면에서 유리하다.

### 2. DB 없이 시작할 경우의 한계

파일 JSON이나 localStorage만으로도 아주 초기 테스트는 가능하다.  
하지만 아래 기능이 들어가면 곧 한계가 온다.

- 날짜별 연습 기록
- 질문별 답변 비교
- 최종 답변 노트 관리
- 콘텐츠 패턴 저장
- 피드백 이력 확인
- 모바일과 PC 간 데이터 이동

따라서 화면만 보는 프로토타입은 무DB로 충분하지만, 실제 개인 사용 MVP는 Turso를 사용한다.

### 3. 초기 데이터 테이블 후보

초기에는 테이블을 많이 만들지 않는다.

- `profile`: 사용자 프로필과 목표 포지션
- `practice_sessions`: 면접 또는 질문 연습 세션
- `practice_turns`: 질문, 사용자 음성 전사, AI 꼬리 질문
- `feedbacks`: 세션 또는 답변별 피드백
- `answer_notes`: 최종 암기용 답변과 핵심 표현
- `daily_patterns`: 매일 생성한 면접영어 패턴 콘텐츠

### 4. 오디오 파일 저장

초기에는 오디오 원본 저장을 필수로 하지 않는다.

우선 저장할 것:

- 질문 텍스트
- 사용자 답변 전사 텍스트
- AI 피드백
- 최종 답변
- 핵심 표현

오디오 파일 저장은 용량, 개인정보, 백업 문제가 있으므로 나중에 결정한다.  
필요하면 파일 스토리지에 오디오를 저장하고 Turso에는 파일 경로나 메타데이터만 저장한다.

### 5. 추천 기술 방향

개인용 모바일 웹/PWA MVP 기준 추천:

- Frontend: 모바일 우선 웹 UI
- Backend: 가벼운 로컬/개인 서버
- DB: Turso 우선
- 음성 면접: OpenAI Realtime API 우선 검토
- 콘텐츠 생성: OpenAI와 Claude API 비교

처음부터 복잡한 클라우드 DB, 로그인, 사용자 관리, 결제, 동기화는 만들지 않는다.

### 6. 모바일에서 Turso 사용 기준

주요 사용 환경은 핸드폰 또는 태블릿이다. 모바일 웹/PWA에서는 기기 안에 DB를 직접 두기보다 백엔드가 Turso에 연결하는 방식을 우선한다.

추천 기준:

- 모바일 브라우저는 직접 DB에 연결하지 않는다.
- 모바일 브라우저는 개인용 백엔드 API를 호출한다.
- 백엔드는 Turso에 연결한다.
- Turso 토큰과 API 키는 백엔드에만 둔다.
- 모바일에는 OpenAI, Claude, Turso 비밀 키를 노출하지 않는다.

대안:

- 네이티브 앱 또는 Capacitor 같은 앱 래퍼를 쓴다면 기기 내부 SQLite/libSQL 사용을 검토할 수 있다.
- 완전 오프라인 PWA를 목표로 한다면 IndexedDB 또는 sqlite-wasm/OPFS 같은 선택지도 있으나, 초기 MVP에는 복잡하다.
- 로컬 개발이나 백업 용도로 순수 SQLite 파일을 병행 사용할 수는 있다.

현재 추천:

- 1차 MVP: 모바일 웹/PWA + 개인용 백엔드 + Turso
- 모바일 기기에는 DB를 직접 두지 않는다.
- 모바일에서는 음성 면접 UI와 결과 조회에 집중한다.
- 데이터 백업과 이전을 고려해 주요 테이블은 단순한 SQL 구조로 유지한다.
