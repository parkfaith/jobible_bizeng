# jobible_bizeng / 조바이블 비즈영어 기획 메모

이 파일은 `jobible_bizeng` 앱 기획을 다른 컴퓨터나 새 AI 세션에서도 이어가기 위한 기준 문서이다.

새 세션에서는 반드시 이 파일을 먼저 읽고 진행한다.  
기획이 바뀌면 항상 이 파일을 업데이트한다.  
현재는 제품 기획 단계이며, 기능 구현은 아직 하지 않는다.

문서와 화면 예시 파일은 모두 UTF-8 인코딩으로 유지한다.  
PowerShell에서 한글이 깨져 보이면 파일 자체 문제로 단정하지 말고, UTF-8로 읽히는지 먼저 확인한다.

프로젝트 저장소:

- https://github.com/parkfaith/jobible_bizeng.git

## 1. 앱 한 줄 정의

`Jobible BizEng`는 외국계 이직을 준비하는 시니어 AI/IT 리더가 모바일에서 AI와 음성으로 실전 비즈니스 영어를 훈련하는 파트너이다. 채용 면접뿐 아니라 임원 보고·부서간 설득·글로벌 팀 피드백 등 실무 시나리오를 통해 영어 말하기 실력을 지속적으로 강화한다.

## 2. 핵심 사용자

대표 사용자: `ryanPark`

- 50세
- AI 기술이사
- 국내 기업에서 20년 이상 근무

- 일상 영어는 어느 정도 가능하지만, 영어 면접에서 경력/성과/리더십을 구조적으로 설명하는 데 약함

## 3. 해결하려는 문제

사용자는 실력과 경력은 충분하지만 영어 면접에서 아래 문제를 겪는다.

- 자기소개와 경력 설명이 길고 흐릿해짐
- 리더십 경험과 프로젝트 성과를 설득력 있게 말하지 못함
- 기술 설명을 영어로 할 때 구조가 무너짐
- 꼬리 질문과 압박 질문에 약함
- 면접처럼 이어지는 영어 대화 연습이 부족함

## 4. 현재 제품 방향

당분간 제품을 넓히지 않는다.

초기 목표는 복잡한 영어 학습 플랫폼이 아니라 `모바일 기반 외국계 영어 면접 5~10분 음성 훈련 앱`이다.

가장 중요한 사용 흐름은 다음 하나다.

1. 오늘의 면접 질문을 받는다.
2. 영어로 말해서 답변한다.
3. AI 면접관이 꼬리 질문을 한다.
4. 답변 피드백을 받는다.
5. 개선 답변을 보고 다시 말한다.
6. 좋은 답변과 표현을 내 답변 노트에 저장한다.

기획이 복잡해지면 이 기본 루프로 돌아온다.

## 5. MVP 핵심 기능

초기 MVP는 아래 5개만 만든다고 가정한다.

### 1. 간단한 프로필 입력

긴 이력서를 받지 않는다. 처음에는 아래만 입력받는다.

- 목표 포지션
- 현재 역할과 경력 연수
- 대표 프로젝트 1~2개
- 가장 걱정되는 면접 상황
- 연습 목표: 자기소개, 경력 설명, 리더십, 기술 설명 중 선택

### 2. 오늘의 질문 연습

짧게 한 문항을 연습하는 모드이다.

- 질문 1개 제시
- 사용자가 영어로 답변
- AI가 꼬리 질문 1개 제시
- 답변 피드백 제공
- 개선 답변을 보고 다시 말하기

### 3. 실전 면접 대화 모드

실제 면접처럼 AI 면접관과 영어로 대화를 이어가는 모드이다. MVP에 반드시 포함한다.  
이 모드는 텍스트 채팅이 아니라 `모바일 음성 대화 중심`으로 설계한다.

- `실전 면접 시작` 버튼을 누르면 먼저 오늘의 면접 유형과 예상 질문 방향을 보여준다.
- 사용자가 안내를 확인한 뒤 `면접 시작`을 눌러 실제 음성 연결을 시작한다.
- AI 면접관이 음성으로 인사 후 첫 질문
- 사용자는 영어로 말해서 답변
- 사용자의 답변은 자동으로 텍스트 변환되어 참고용으로 화면에 표시
- AI 면접관은 사용자의 답변에 따라 자연스럽게 다음 질문 또는 꼬리 질문
- 한 세션은 3~4개 질문, 5~10분으로 제한
- 질문 흐름은 오늘의 답변 패턴세트 주제를 중심으로 진행하고, 필요하면 경력/리더십/기술/갈등으로 자연스럽게 확장한다.
- 면접 중에는 코칭하지 않고 실제 면접처럼 진행
- 면접 종료 후 전체 피드백 제공
- 면접 중 사용자가 직접 타이핑하는 UI는 두지 않는다
- 텍스트는 입력 수단이 아니라 참고 자료, 음성 인식 결과, 개선 답변 확인용으로만 사용한다

초기에는 면접관 캐릭터를 여러 개 만들지 않는다.  
`외국계 AI Director 면접관` 1명만 사용한다.

### 4. 답변 코칭

피드백은 많을수록 좋은 것이 아니다. 사용자가 바로 고칠 수 있게 짧고 실용적으로 준다.

일반 답변 피드백:

- 핵심 메시지가 분명한가
- 답변 구조가 좋은가
- 더 자연스러운 영어 표현
- 시니어 리더다운 개선 답변 예시

실전 면접 종료 피드백:

- 가장 좋았던 답변 1개
- 가장 위험했던 답변 1개
- 다음 연습에서 반드시 고칠 점 1개
- 면접에서 바로 쓸 수 있는 개선 문장 3개

### 5. 내 답변 노트

면접 전에 다시 볼 수 있는 답변 저장소이다.

- 자기소개 답변
- 경력 설명 답변
- 리더십 사례 답변
- 기술 프로젝트 설명 답변
- 실패/갈등 해결 답변
- 바로 쓸 수 있는 핵심 영어 문장

## 6. MVP 화면

화면은 3개로 충분하다.

초기 화면은 모바일을 기본 기준으로 설계한다.  
이 앱은 출퇴근 전후, 점심시간, 면접 직전처럼 짧은 시간에 쓰일 가능성이 높으므로 데스크톱보다 모바일 사용성이 중요하다. 데스크톱 화면은 나중에 확장한다.

### 1. 홈

- 오늘의 질문 연습 시작
- 음성 실전 면접 시작
- 최근 저장한 답변
- 이번 주 연습 횟수

### 2. 연습 화면

- AI 면접관 음성 질문
- 큰 마이크 버튼
- 녹음 상태와 남은 시간
- 음성 인식된 답변 텍스트
- 참고용 답변 구조 또는 핵심 표현
- 면접 후 피드백
- 다시 말하기

`오늘의 질문 연습`과 `실전 면접 대화 모드`는 같은 연습 화면 안에서 처리한다.

### 3. 답변 노트

- 저장된 답변
- 저장된 핵심 표현
- 면접 전 빠르게 다시 볼 답변 모음

## 7. 모바일 사용 방향

모바일을 기본 사용 환경으로 본다. 다만 처음부터 네이티브 앱을 만들기보다 `모바일 웹` 또는 `PWA`로 시작하는 것이 현실적이다.

모바일에서 특히 중요한 점:

- 큰 `음성 실전 면접 시작` 버튼
- 한 손으로 누르기 쉬운 녹음/일시정지/제출 버튼
- AI 질문은 짧게 표시하고 음성으로 들려주기
- 사용자의 음성 인식 텍스트는 참고용으로 크게 보여주기
- 타이핑 입력 영역은 두지 않기
- 공부를 돕는 참고 내용은 읽기 전용으로 제공하기
- 면접 중에는 화면 조작을 최소화하기
- 조용한 공간이 아니어도 이어폰으로 연습할 수 있게 고려하기
- 녹음 권한, 마이크 상태, 네트워크 상태를 명확히 보여주기

모바일에서 당장 복잡하게 만들 필요는 없다.  
초기 모바일 경험은 `버튼 누르기 -> 듣기 -> 말하기 -> 음성 인식 결과 확인 -> 다음 질문` 흐름만 자연스러우면 충분하다.

## 8. 초기에는 제외할 기능

좋은 아이디어라도 초기에는 아래 기능을 넣지 않는다.

- 모든 비즈니스 업무 상황 훈련
- 회의, 협상, 고객 커뮤니케이션 전체 시나리오
- 정교한 발음 점수화
- 학습 레벨, 배지, 랭킹, 게임화
- 커뮤니티
- 채용공고 검색
- 이력서/LinkedIn 전체 자동 작성
- 여러 AI 면접관 캐릭터
- 과도하게 세분화된 진도 관리

단, `실전 면접 대화 모드`는 제외하지 않는다. 이 앱의 핵심 경험이다.

## 9. AI 코치의 말투와 역할

AI는 친절한 영어 선생님보다 `실전 면접 코치이자 글로벌 커리어 멘토`처럼 동작해야 한다.

- 지나치게 귀엽거나 가벼운 말투는 피한다.
- 답변을 무조건 칭찬하지 않는다.
- 시니어 리더에게 맞는 간결하고 설득력 있는 표현을 제안한다.
- 면접 중에는 실제 면접관처럼 질문한다.
- 코칭은 면접이 끝난 뒤에 한다.

## 10. 기획 검토 및 수정 제안

현재 기획은 MVP로 적절하다. 다만 아래 사항은 다음 기획에서 반드시 더 구체화한다.

### 1. 실전 면접 대화 모드를 중심 기능으로 둔다

`오늘의 질문 연습`은 가벼운 워밍업이고, 핵심 경험은 `실전 면접 대화 모드`이다.  
홈 화면에서도 `음성 실전 면접 시작`을 가장 중요한 행동으로 배치한다.

### 2. 프로필 입력은 경력 사실 수집에 집중한다

AI가 사용자의 경력을 지어내면 안 된다. 온보딩에서는 멋진 문장보다 실제 사실을 짧게 수집한다.

- 대표 프로젝트명
- 맡은 역할
- 해결한 문제
- 사용한 기술 또는 방법
- 성과 수치 또는 비즈니스 임팩트
- 리더십/갈등/실패 사례 1개

### 3. 면접은 영어로, 코칭은 한국어 중심으로 한다

실전 면접 대화는 영어로 진행한다.  
면접 후 피드백은 사용자가 빠르게 이해할 수 있도록 한국어 중심으로 제공하고, 개선 문장은 영어로 제공한다.

### 4. 피드백 기준을 단순한 4개 항목으로 고정한다

답변 평가는 아래 4개만 본다.

- 내용: 질문에 제대로 답했는가
- 구조: 결론, 근거, 사례, 성과가 보이는가
- 영어: 표현이 자연스럽고 간결한가
- 리더십 톤: 시니어 리더답게 들리는가

점수화가 필요하다면 항목별 5점 척도까지만 사용한다.

### 5. 답변 노트는 최종 답변 저장소로 만든다

답변 노트에는 단순 대화 기록이 아니라 면접 전에 다시 볼 수 있는 최종본을 저장한다.

- 원래 답변
- 개선된 답변
- 최종 암기용 답변
- 바로 쓸 핵심 표현 2~3개

### 6. 압박 질문은 별도 기능으로 만들지 않는다

초기에는 `압박 질문 모드`를 따로 만들지 않는다.  
실전 면접 대화의 후반부에 자연스러운 꼬리 질문이나 검증 질문을 1~2개 넣는 정도로 충분하다.

### 7. 실전 면접은 음성 우선으로 설계한다

이 앱은 면접 훈련 앱이므로 실전 면접 대화는 음성 우선이어야 한다.

- AI 면접관 질문은 음성으로 들려준다.
- 사용자는 영어로 말해서 답변한다.
- 화면에는 음성 인식된 답변 텍스트를 함께 보여준다.
- 음성 인식 결과는 확인용이며, 면접 중 직접 수정하지 않는다.
- 피드백은 면접이 끝난 뒤 제공한다.
- 면접 중 타이핑은 하지 않는다.

초기에는 정교한 발음 점수화나 텍스트 작성 기능보다 `말문이 막히지 않고 면접 흐름을 이어가는 경험`이 더 중요하다.  
구현 단계에서는 최신 공식 OpenAI 문서를 확인한 뒤 음성 입력, 실시간 음성 대화, 녹음 기반 피드백 중 MVP에 맞는 방식을 결정한다.

### 8. 성공 기준을 작게 잡는다

초기 MVP의 성공은 많은 기능이 아니라 아래 행동이 반복되는지로 판단한다.

- 사용자가 실전 면접을 끝까지 완료한다.
- 사용자가 개선 답변을 다시 말한다.
- 사용자가 답변 노트에 최종 답변을 저장한다.
- 일주일에 3회 이상 연습할 이유가 생긴다.

현재 화면 예시는 아래 3개 화면을 포함한다.

- 홈: 오늘의 질문, 음성 실전 면접 시작, 최근 답변, 프로필 요약
- 연습 화면: 모바일 음성 면접, 답변 텍스트 변환, 참고 표현, 면접 후 피드백 예시
- 답변 노트: 원래 답변, 개선 답변, 최종 암기용 답변, 핵심 표현

다음 검토에서는 아래를 확인한다.

- 홈에서 `음성 실전 면접 시작`이 충분히 중심 행동으로 보이는가
- 연습 화면이 실제 음성 면접처럼 긴장감 있게 느껴지는가
- 타이핑 없이도 연습 흐름이 자연스러운가
- 피드백이 너무 많거나 학습 앱처럼 느껴지지 않는가
- 답변 노트가 면접 전날 다시 보기 좋은 구조인가

## 13. API 사용 방향 초안

현재 판단은 아래와 같다.

### 1. 사용자-facing 음성 면접

모바일에서 실제 면접처럼 음성으로 대화하는 핵심 기능은 OpenAI Realtime API 계열을 우선 검토한다.

이유:

- 실시간 음성 입출력에 맞는 구조이다.
- 모바일 웹/PWA에서 WebRTC 기반 연결을 검토할 수 있다.
- 사용자의 말투, 끊김, 답변 흐름을 살린 면접 경험에 적합하다.

### 2. 면접 후 피드백과 답변 노트

면접이 끝난 뒤에는 음성 대화 기록 또는 전사 텍스트를 바탕으로 별도 텍스트 모델이 피드백을 생성한다.

피드백 생성 항목:

- 내용
- 구조
- 영어 표현
- 리더십 톤
- 최종 암기용 답변
- 바로 쓸 핵심 표현

### 3. 매일 제공할 면접영어 패턴 콘텐츠

매일 받는 면접영어 패턴, 질문 세트, 모범 답변, 핵심 표현은 실시간 음성 API가 아니라 텍스트 생성 API로 만드는 것이 적합하다.

이 영역은 Anthropic Claude API도 적극 비교한다.  
Claude Code 자체는 개발 보조 도구이지만, Claude 계열 모델은 긴 글 정리, 패턴화, 커리큘럼 구성, 문장 다듬기, 사례 기반 콘텐츠 제작에서 강점이 있을 수 있다.

콘텐츠 생성에서 비교할 항목:

- 시니어 리더에게 맞는 자연스러운 비즈니스 영어 표현
- 면접 질문 패턴의 실전성
- 모범 답변의 구조와 설득력
- 한국어 설명의 명확성
- 매일 반복 학습하기 좋은 분량 조절
- 답변 노트에 저장하기 좋은 최종 문장 품질

### 4. 현재 추천

MVP 단계 추천은 `음성은 OpenAI, 콘텐츠는 OpenAI와 Claude 비교` 구조이다.

- 음성 면접: OpenAI Realtime API
- 면접 후 피드백: 우선 OpenAI 텍스트 모델, 이후 Claude와 품질 비교
- 매일 면접영어 패턴 생성: OpenAI와 Claude API를 같은 프롬프트로 비교
- 최종 선택 기준: 비용보다 `콘텐츠 품질`, `일관성`, `운영 단순성`을 우선한다

Claude Code는 개발 보조 도구로 보고, 실제 앱 사용자에게 제공되는 런타임 API로 보지는 않는다.  
앱에서 Claude를 쓴다면 Claude Code가 아니라 Anthropic Claude API를 사용한다.

구현 단계에서는 최신 공식 OpenAI 문서와 Anthropic 문서를 다시 확인한 뒤 모델명, 비용, 지연시간, 모바일 브라우저 지원 방식을 결정한다.

## 14. 축적 루프 확장 (2026-06-11)

"연습은 하는데 늘고 있는지 안 보이고, 피드백이 휘발된다"는 문제를 해결하기 위해
콘텐츠 공급을 늘리지 않고 연습이 쌓이는 구조 3가지를 추가했다.
사용자 추가 부담: 약점 추적 = 0, 마스터 모드 = 버튼 1개, JD 모드 = 공고 붙여넣기 1회.

### 1. 약점 추적 루프

- 면접 피드백에서 반복 약점 2~3개를 고정 태그로 추출해 누적한다.
- 다음 실전 면접에서 면접관이 약점 영역을 자연스럽게 검증한다 (약점이라고 밝히지 않음).
- 면접 후 피드백에 "지난번 지적사항이 개선됐는지" 평가를 포함한다.

### 2. 핵심 답변 마스터 모드

- 답변 노트의 질문을 같은 질문으로 재도전한다. 진입은 답변 노트의 "이 질문 다시 도전" 버튼.
- 직전 시도와 점수 델타·답변 비교·진전 평가를 보여준다. 실력 향상이 눈에 보이는 유일한 장치.
- 새 노트를 만들지 않고 기존 노트의 최종 답변을 갱신한다.

### 3. JD 모드

- 실제 지원할 채용공고를 붙여넣으면 요구사항을 추출해 저장한다 (`/jd`).
- 그 공고 기준 맞춤 실전 면접을 진행하고, 피드백에 요구사항 커버리지(입증/놓침)를 표시한다.
- 이벤트성 기능: 공고가 없으면 UI에 거의 나타나지 않는다.

## 15. 현재 기술 상태 요약

상세 구현 현황, API 라우트 목록, DB 테이블, 배포 판단, 날짜별 작업 로그는 새 세션 기본 문서에서 분리했다.

상세 문서:

- `docs/technical-status.md` — 구현 완료 상태, 기술 스택, API/DB/배포 기준
- `docs/handoff.md` — 날짜별 작업 인계, 디버깅 기록, 실기기 테스트 체크리스트

현재 앱 기준:

- Framework: Next.js App Router
- DB: Turso/libSQL + Drizzle ORM
- Styling: Tailwind CSS
- 음성 면접: OpenAI Realtime API 계열 우선
- 오늘의 패턴/피드백/TTS: OpenAI API 사용
- 배포: Vercel 자동배포, GitHub `master` 푸시 기준
- 비밀값은 `.env.local`과 Vercel 환경변수에만 둔다. 절대 커밋하지 않는다.

새 컴퓨터 또는 새 세션에서 시작할 때:

```bash
# .env.local 없으면 .env.example 참고해서 생성
npm install   # package.json 변경이 있을 때만
npm run dev
```

운영 원칙:

- 제품 방향이나 MVP 범위가 바뀌면 이 `agents.md`의 핵심 섹션을 업데이트한다.
- 구현 상세, 장애 원인 분석, 배포 후 확인 사항은 `docs/technical-status.md` 또는 `docs/handoff.md`에 기록한다.
- 새 세션에서는 먼저 이 문서를 읽고, 최근 작업 맥락이 필요할 때만 `docs/handoff.md`의 최신 섹션을 확인한다.
